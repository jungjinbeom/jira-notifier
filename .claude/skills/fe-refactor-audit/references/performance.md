# 성능 감사 (fe-performance)

jira-notifier 프론트의 렌더링/런타임 성능. 이 앱은 소규모이므로 **측정 가능한 실익이 있는 것만** 제안한다. 근거 없는 선제 최적화 금지.

## 1. 리렌더 범위
- App이 최상위에서 모든 상태(config·notifications·unassigned·myTickets·status)를 보유 → 어느 하나 바뀌면 App 전체 서브트리 리렌더.
- **관찰 포인트**: `status`는 `setInterval(5s)`로 주기 갱신 → 5초마다 App 리렌더 발생. 알림/티켓 이벤트 유입 시에도 전체 리렌더.
- **개선(실익 있을 때)**: 상태를 소비 지점에 가깝게 내리거나(god-hook 분해와 연동), 무거운 목록을 `React.memo`로 격리. **단**, 현재 트리가 얕고 항목 수가 적으면 체감 이득이 없을 수 있음 → "구조 개선의 부수효과"로만 제안하고 단독 최적화는 보류 가능.

## 2. 함수 안정성 (useCallback)
- `refreshUnassigned`/`refreshMyTickets`/`markAsRead` 등이 매 렌더 재생성되어 자식 prop으로 전달.
- **개선**: 자식을 `memo`로 감쌀 경우에 한해 `useCallback`으로 안정화(둘은 세트). memo 없이 useCallback만 추가하면 이득 없이 복잡도만↑ → 함께 판단.

## 3. 이펙트/리스너 정확성
- `listen(...)` 구독들은 빈 의존성 배열로 1회 등록·언마운트 해제 → 대체로 올바름. 확인: 핸들러가 최신 상태를 참조해야 하면 stale closure 위험(setX 함수형 업데이트로 회피되는지 점검).
- `setInterval(loadStatus, 5000)`: 폴링과 별개로 5초 상태 폴링이 필요한 주기인지 재검토(백엔드가 이벤트로 밀어줄 수 있으면 폴링 축소 가능 — 단 이는 동작 변경일 수 있으니 감독자 판단).

## 4. 번들/임포트
- 사용처 없는 의존/무거운 임포트 확인. 현재 의존은 react/react-dom + @tauri-apps/*로 가벼움 → 큰 이슈 없을 가능성. 확인 후 "이상 없음"도 유효한 결론.

## 판단 기준
- 대부분의 성능 항목은 **구조 리팩토링(god-hook 분해·memo 격리)의 부수효과**로 얻는 것이 안전. 가독성을 해치는 단독 미세 최적화는 근거(리렌더 빈도·목록 크기) 없이는 제안하지 않는다.
- "최적화 불필요"라는 결론도 정당한 산출물이다.
